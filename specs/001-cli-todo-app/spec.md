# Feature Specification: CLI 기반 ToDo 관리 앱

**Feature Branch**: `001-cli-todo-app`  
**Created**: 2026-05-02  
**Status**: Draft  
**Input**: CLI 기반의 ToDo 관리 앱. 터미널을 사용하는 개인 개발자가 항목 추가, 목록 조회, 완료 처리, 삭제를 명령행에서 수행한다.

## Constitution Alignment *(mandatory)*

- **CA-001 (레이어 분리)**: 항목 추가/조회/완료/삭제 비즈니스 규칙은 CLI 입력 파싱·출력 포맷팅과 분리된 독립 레이어에서 처리된다. CLI 레이어는 사용자 입력을 도메인 요청으로 변환하고 결과를 화면에 출력하는 역할만 담당한다.
- **CA-002 (테스트 우선)**: 각 user story는 구현 전에 실패 상태의 테스트를 먼저 작성한다. 항목 추가·조회·완료·삭제 각각에 대한 단위 테스트와 CLI 통합 테스트가 선행된다.
- **CA-003 (최소 의존성)**: 런타임 의존성은 **Typer**, **SQLAlchemy**만 사용한다. 데이터 저장은 로컬 **SQLite** 파일(`todo.db`)로 한정하며, 이외 패키지는 명확한 필요성이 입증되지 않는 한 도입하지 않는다.
- **CA-004 (단순함 우선)**: 4개 기능(추가·조회·완료·삭제) 각각을 단순하고 직접적으로 구현한다. 플러그인 시스템, ORM, 이벤트 버스 등 현재 필요하지 않은 추상화는 도입하지 않는다.
- **CA-005 (CLI 범위)**: REST API 서버, GUI, 웹 인터페이스는 이 프로젝트의 범위 밖이다. 모든 사용자 상호작용은 명령행과 표준 입출력으로만 이루어진다.

## Scope Boundaries *(mandatory)*

- In Scope: 터미널 CLI 기반 ToDo 항목 추가·조회·완료 처리·삭제
- Out of Scope: REST API 서버, GUI 앱, 웹 인터페이스, 다중 사용자 동기화, 알림/리마인더 기능

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ToDo 항목 추가 (Priority: P1)

터미널을 사용하는 개인 개발자가 명령행에서 새로운 할 일을 등록한다.
제목은 필수이고, 마감일과 우선순위는 선택적으로 입력할 수 있다.
항목이 성공적으로 추가되면 새로 부여된 ID와 함께 확인 메시지가 출력된다.
앱을 재실행해도 추가된 항목이 유지된다.

**Why this priority**: 항목 추가 기능이 없으면 나머지 모든 기능(조회·완료·삭제)이 의미 없다.
가장 먼저 동작해야 하는 핵심 기능이다.

**Independent Test**: 새 항목을 추가하는 명령을 실행한 뒤, 저장소에서 해당 항목 데이터를 조회해 제목·마감일·우선순위·완료 여부가 올바르게 저장되었는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 저장된 항목이 없는 초기 상태, **When** 제목만 입력해 추가 명령을 실행하면, **Then** ID 1이 부여된 항목이 저장되고 "항목이 추가되었습니다 (ID: 1)" 형태의 확인 메시지가 출력된다.
2. **Given** 이미 항목이 있는 상태, **When** 제목·마감일·우선순위를 모두 입력해 추가 명령을 실행하면, **Then** 새 고유 ID가 부여된 항목이 저장되고 확인 메시지에 ID가 포함된다.
3. **Given** 어떤 상태든, **When** 제목 없이 추가 명령을 실행하면, **Then** "제목은 필수 입력 항목입니다" 오류 메시지가 출력되고 항목은 저장되지 않는다.
4. **Given** 항목을 추가한 뒤 앱을 재시작, **When** 목록을 조회하면, **Then** 이전에 추가한 항목이 그대로 표시된다.

---

### User Story 2 - 전체 목록 조회 및 필터링 (Priority: P2)

개발자가 명령행에서 저장된 ToDo 항목 전체를 조회하고,
완료 여부 또는 우선순위 기준으로 필터링해 필요한 항목만 볼 수 있다.
각 항목은 ID·제목·마감일·우선순위·완료 여부를 포함한 형태로 표시된다.

**Why this priority**: 항목 추가 후 목록을 확인할 수 없으면 도구로서 가치가 없다.
항목 추가(P1) 다음으로 가장 빈번하게 사용되는 기능이다.

**Independent Test**: 여러 항목(완료/미완료, 우선순위 혼합)을 미리 저장한 뒤
필터 없는 전체 조회와 각 필터 조회를 실행해 출력 결과를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 완료·미완료 항목이 혼합된 상태, **When** 필터 없이 목록 조회 명령을 실행하면, **Then** 저장된 모든 항목이 ID·제목·마감일·우선순위·완료 여부와 함께 출력된다.
2. **Given** 완료·미완료 항목이 혼합된 상태, **When** 미완료 필터로 조회하면, **Then** 완료 처리된 항목은 결과에서 제외되고 미완료 항목만 출력된다.
3. **Given** 완료·미완료 항목이 혼합된 상태, **When** 완료 필터로 조회하면, **Then** 완료 처리된 항목만 출력된다.
4. **Given** 우선순위가 다른 항목들이 있는 상태, **When** 특정 우선순위 필터로 조회하면, **Then** 해당 우선순위에 해당하는 항목만 출력된다.
5. **Given** 저장된 항목이 없는 상태, **When** 목록 조회 명령을 실행하면, **Then** "등록된 항목이 없습니다" 메시지가 출력된다.

---

### User Story 3 - 항목 완료 처리 (Priority: P3)

개발자가 완료한 할 일의 ID를 입력해 해당 항목을 완료 상태로 변경한다.
완료 처리 후 해당 항목이 완료 상태로 업데이트됐음을 확인하는 메시지가 출력된다.

**Why this priority**: 할 일의 생명 주기에서 핵심 상태 전환이다.
추가(P1)·조회(P2) 이후 실제 작업 흐름에서 가장 많이 쓰이는 기능이다.

**Independent Test**: 미완료 항목을 저장한 뒤 완료 처리 명령을 실행하고,
저장소에서 해당 항목의 완료 여부가 변경됐는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 미완료 상태의 항목이 존재할 때, **When** 해당 항목의 ID로 완료 처리 명령을 실행하면, **Then** 항목이 완료 상태로 변경되고 "항목 [ID]가 완료 처리되었습니다" 메시지가 출력된다.
2. **Given** 이미 완료된 항목에 대해, **When** 동일 ID로 완료 처리 명령을 실행하면, **Then** "항목 [ID]는 이미 완료된 항목입니다" 안내 메시지가 출력되고 상태는 변경되지 않는다.
3. **Given** 존재하지 않는 ID로, **When** 완료 처리 명령을 실행하면, **Then** "항목 [ID]를 찾을 수 없습니다" 오류 메시지가 출력된다.

---

### User Story 4 - 항목 삭제 (Priority: P4)

개발자가 더 이상 필요 없는 ToDo 항목을 ID로 지정해 영구 삭제한다.
삭제 후 해당 항목은 목록에서 제거되며 복구할 수 없다.

**Why this priority**: 잘못 입력한 항목이나 불필요한 항목 정리를 위한 기능이다.
기본 CRUD의 마지막 단계로, 앞선 세 기능보다 사용 빈도가 낮다.

**Independent Test**: 항목을 저장한 뒤 삭제 명령을 실행하고,
목록 조회 시 해당 항목이 더 이상 나타나지 않는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 특정 ID의 항목이 존재할 때, **When** 해당 ID로 삭제 명령을 실행하면, **Then** 항목이 저장소에서 제거되고 "항목 [ID]가 삭제되었습니다" 메시지가 출력된다.
2. **Given** 존재하지 않는 ID로, **When** 삭제 명령을 실행하면, **Then** "항목 [ID]를 찾을 수 없습니다" 오류 메시지가 출력된다.
3. **Given** 항목을 삭제한 뒤, **When** 목록을 조회하면, **Then** 삭제된 항목은 결과에 나타나지 않는다.

---

### Edge Cases

- 빈 문자열이나 공백만으로 구성된 제목을 입력하면 어떻게 처리하는가?
- 과거 날짜를 마감일로 입력하면 "마감일이 과거 날짜입니다" 경고를 출력하고 그대로 저장한다. (저장 허용, 거부 없음)
- 유효하지 않은 날짜 형식(예: "내일", "20260532")을 입력하면 어떻게 처리하는가?
- ID 입력란에 숫자가 아닌 값을 입력하면 어떻게 처리하는가?
- 저장 파일이 외부에서 손상된 경우 "데이터 파일이 손상되었습니다. 파일을 백업하거나 삭제 후 재시도하세요" 오류 메시지를 출력하고 종료한다. (자동 복구 없음)
- 동시에 같은 데이터 파일을 두 개의 터미널에서 접근하면 어떻게 처리하는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 제목(필수), 마감일(선택), 우선순위(선택)를 지정해 새 ToDo 항목을 추가할 수 있어야 한다.
- **FR-002**: 항목 추가 시 고유 ID가 자동으로 부여되어야 한다.
- **FR-003**: 추가된 항목은 앱 종료 후 재실행해도 유지되어야 한다. 데이터는 로컬 SQLite 파일(`todo.db`)에 저장된다.
- **FR-004**: 사용자는 저장된 모든 ToDo 항목 목록을 조회할 수 있어야 한다.
- **FR-005**: 사용자는 완료 여부(완료/미완료) 기준으로 목록을 필터링할 수 있어야 한다.
- **FR-006**: 사용자는 우선순위 기준으로 목록을 필터링할 수 있어야 한다.
- **FR-007**: 목록 출력 시 각 항목의 ID·제목·마감일·우선순위·완료 여부가 표시되어야 한다.
- **FR-008**: 사용자는 항목 ID를 이용해 특정 항목을 완료 상태로 변경할 수 있어야 한다.
- **FR-009**: 사용자는 항목 ID를 이용해 특정 항목을 영구 삭제할 수 있어야 한다.
- **FR-010**: 존재하지 않는 ID에 대한 완료 처리 또는 삭제 시도 시 명확한 오류 메시지를 출력해야 한다.
- **FR-011**: 제목이 빈 값이거나 공백만인 경우 항목 추가를 거부하고 오류 메시지를 출력해야 한다.
- **FR-012**: 유효하지 않은 날짜 형식 입력 시 오류 메시지를 출력하고 저장하지 않아야 한다.
- **FR-014**: 과거 날짜를 마감일로 입력하면 "마감일이 과거 날짜입니다" 경고를 출력하고 항목을 저장한다.
- **FR-015**: 데이터 파일(`todo.db`) 접근 또는 조회가 실패하면 "데이터 파일이 손상되었습니다. 파일을 백업하거나 삭제 후 재시도하세요" 메시지를 출력하고 종료한다.
- **FR-013**: 우선순위는 high / medium / low 세 단계로 구분한다.

### Key Entities

- **ToDo 항목**: 고유 ID(자동 부여), 제목(문자열, 필수), 마감일(날짜, 선택), 우선순위(high/medium/low, 선택), 완료 여부(boolean), 생성 일시

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 항목 추가·조회·완료·삭제를 각각 단일 명령으로 30초 이내에 완료할 수 있다.
- **SC-002**: 앱 재실행 후에도 이전에 저장한 모든 항목이 손실 없이 유지된다.
- **SC-003**: 잘못된 입력(빈 제목, 잘못된 날짜, 존재하지 않는 ID) 시 100% 의 경우에서 명확한 오류 메시지가 출력되고 데이터가 손상되지 않는다.
- **SC-004**: 필터 조회 결과가 항상 필터 조건과 일치한다 (필터링 정확도 100%).
- **SC-005**: 1,000개 이상의 항목이 저장된 상태에서 모든 기능이 체감상 즉각적으로 응답한다.

- Q: CLI 라이브러리는 무엇을 사용하는가? → A: Typer (`pip install typer`, type hint 기반 서브커맨드)

- Q: 저장 파일 손상 시 처리 방식은? → A: 오류 메시지 출력 후 종료
- Q: 크로스 플랫폼 지원 범위는? → A: Windows, macOS, Linux 모두 지원

## Assumptions

- 단일 사용자가 단일 머신에서 사용하는 로컬 전용 도구다. 다중 사용자 동기화는 범위 밖이다.
- 항목 데이터는 로컬 **SQLite 파일**(`todo.db`)에 저장된다. ORM은 **SQLAlchemy**를 사용하며, 외부 데이터베이스 서버나 클라우드 연동은 범위 밖이다.
- 우선순위는 high / medium / low 세 단계로 한정한다. 사용자 정의 우선순위는 범위 밖이다.
- 마감일 입력 형식은 YYYY-MM-DD 를 표준으로 한다.
- 항목 ID는 앱이 자동으로 부여하며, 사용자가 직접 지정하지 않는다.
- 앱은 인터랙티브 셸이 아닌 서브커맨드 방식(예: `todo add "제목"`)으로 동작한다. CLI 구현에는 **Typer** 라이브러리를 사용한다 (`pip install typer`).
- 운영 환경은 Windows(PowerShell/CMD), macOS, Linux 모두 지원한다. Typer와 Python 표준 라이브러리 조합으로 별도 플랫폼별 분기 없이 크로스 플랫폼을 지원한다.
