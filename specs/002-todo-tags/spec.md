# Feature Specification: ToDo 태그 기능 추가

**Feature Branch**: `002-todo-tags`  
**Created**: 2026-05-03  
**Status**: Draft  
**Input**: User description: "기존 Todo 앱에 태그 기능을 추가하고 싶다. Todo 항목 생성 시 태그를 선택적으로 추가할 수 있고, 태그는 최대 5개까지 허용되며 각 태그는 20자 이내다. 태그 없이도 Todo 생성이 가능해야 한다. 목록 조회 시 `todo list --tag 태그명` 형식으로 태그 필터링이 가능해야 한다. 기존 Todo 기능(추가/조회/완료/삭제)은 그대로 유지되어야 한다. 관련 이슈 번호는 #1, #2다." 

## Clarifications

### Session 2026-05-03

- Q: ToDo 생성 시 태그 입력 문법은 무엇으로 고정할까? → A: `--tag` 옵션 반복 입력 방식 사용
- Q: 태그 대소문자 처리 기준은 무엇으로 할까? → A: 비교는 대소문자 비구분, 저장/표시는 최초 입력 원형 유지
- Q: 태그 허용 문자 집합은 무엇으로 할까? → A: 영문/숫자/-/_만 허용, 공백 금지

## Constitution Alignment *(mandatory)*

- **CA-001 (레이어 분리)**: 태그의 유효성 검사, 저장, 조회 조건 적용과 같은 비즈니스 규칙은 기존 ToDo 도메인 레이어에서 처리하고, CLI 레이어는 입력 전달과 결과 출력만 담당한다.
- **CA-002 (테스트 우선)**: 태그 추가, 태그 필터 조회, 기존 기능 회귀 방지는 각각 실패하는 테스트를 먼저 작성한 뒤 구현한다.
- **CA-003 (최소 의존성)**: 이 기능은 기존 프로젝트의 의존성 범위 안에서 해결하며, 태그 기능만을 위해 새로운 외부 의존성을 추가하지 않는다.
- **CA-004 (단순함 우선)**: 태그는 기존 ToDo 항목에 직접 연결되는 단순 속성으로 취급하며, 별도 권한 체계나 복잡한 분류 체계는 도입하지 않는다.
- **CA-005 (CLI 범위)**: 태그 기능은 기존 CLI 명령 흐름 안에서만 제공되며, API, GUI, 웹 인터페이스는 범위 밖이다.

## Scope Boundaries *(mandatory)*

- In Scope: ToDo 항목 생성 시 선택적 태그 입력, 태그 유효성 검사, 태그 기준 목록 필터링, 기존 기능 유지
- Out of Scope: 태그 색상/설명/계층 구조, 다중 태그 조건 검색, 태그 일괄 편집, API 또는 GUI 제공

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 태그가 있는 ToDo 생성 (Priority: P1)

개인 개발자가 새로운 ToDo를 만들 때 작업 성격을 구분하기 위해 태그를 함께 기록한다. 태그는 선택 사항이므로 필요할 때만 입력할 수 있으며, 허용 범위를 벗어난 태그는 저장되기 전에 거부된다.

**Why this priority**: 태그 필터링은 태그가 저장되어 있어야 의미가 있으므로, 태그 부여가 가장 먼저 제공되어야 하는 핵심 흐름이다.

**Independent Test**: 사용자가 ToDo를 생성하면서 태그를 함께 입력했을 때 항목이 저장되고, 저장된 태그가 그대로 조회 가능한지 확인하면 이 사용자 가치를 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 새 ToDo를 생성하려는 상태, **When** 제목과 함께 `--tag` 옵션을 1회 이상 반복해 유효한 태그를 입력하면, **Then** 항목이 저장되고 입력한 태그가 해당 항목에 연결된다.
2. **Given** 사용자가 새 ToDo를 생성하려는 상태, **When** 태그를 입력하지 않으면, **Then** 항목은 기존과 동일하게 정상 저장된다.
3. **Given** 사용자가 새 ToDo를 생성하려는 상태, **When** 태그를 6개 이상 입력하면, **Then** 항목 저장이 거부되고 태그는 최대 5개까지만 허용된다는 안내가 표시된다.
4. **Given** 사용자가 새 ToDo를 생성하려는 상태, **When** 20자를 초과하는 태그를 입력하면, **Then** 항목 저장이 거부되고 각 태그는 20자 이내여야 한다는 안내가 표시된다.
5. **Given** 사용자가 새 ToDo를 생성하려는 상태, **When** 공백이나 대소문자 차이만 있는 중복 태그를 함께 입력하면, **Then** 항목 저장이 거부되고 같은 항목에는 중복 태그를 둘 수 없다는 안내가 표시된다.

---

### User Story 2 - 태그로 목록 필터링 (Priority: P2)

개인 개발자가 많은 ToDo 중 특정 주제나 작업 묶음만 빠르게 보고 싶을 때 태그를 지정해 목록을 좁혀 본다. 사용자는 태그 하나를 기준으로 원하는 항목만 확인할 수 있어야 한다.

**Why this priority**: 태그의 실질적인 사용자 가치는 필요한 항목을 빠르게 찾는 데 있으므로, 저장 다음으로 중요한 사용 흐름이다.

**Independent Test**: 태그가 서로 다른 항목을 여러 개 저장한 뒤 특정 태그로 목록을 조회해 해당 태그가 연결된 항목만 나타나는지 확인하면 이 흐름을 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 여러 ToDo에 서로 다른 태그가 저장된 상태, **When** 사용자가 특정 태그로 목록을 조회하면, **Then** 해당 태그를 가진 항목만 결과에 포함된다.
2. **Given** 태그가 없는 항목과 태그가 있는 항목이 함께 존재하는 상태, **When** 사용자가 특정 태그로 목록을 조회하면, **Then** 태그가 없는 항목은 결과에 포함되지 않는다.
3. **Given** 어떤 항목도 요청한 태그를 가지지 않는 상태, **When** 사용자가 해당 태그로 목록을 조회하면, **Then** 조건에 맞는 항목이 없다는 안내가 표시된다.

---

### User Story 3 - 기존 ToDo 흐름 유지 (Priority: P3)

기존 사용자는 태그 기능이 추가된 뒤에도 이전과 같은 방식으로 ToDo를 추가, 조회, 완료, 삭제할 수 있어야 한다. 태그를 사용하지 않는 사용자도 기능 변화 없이 기존 흐름을 그대로 유지해야 한다.

**Why this priority**: 새 기능이 기존 핵심 기능을 깨뜨리면 사용자 신뢰와 학습 비용에 직접적인 문제가 생기므로 회귀 방지는 필수다.

**Independent Test**: 태그를 사용하지 않는 항목에 대해 기존 추가, 전체 조회, 완료, 삭제 흐름을 그대로 실행해 결과가 이전과 동일한지 확인하면 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 태그 없이 ToDo를 생성하는 상태, **When** 기존 방식으로 항목을 추가하면, **Then** 이전과 동일한 성공 흐름으로 저장된다.
2. **Given** 태그가 없는 항목과 태그가 있는 항목이 함께 존재하는 상태, **When** 사용자가 필터 없이 전체 목록을 조회하면, **Then** 모든 항목이 기존 조회 규칙대로 표시된다.
3. **Given** 태그 유무가 섞인 항목들이 존재하는 상태, **When** 사용자가 기존 방식으로 완료 또는 삭제를 수행하면, **Then** 대상 항목만 정상 처리되고 다른 동작에는 영향을 주지 않는다.

---

### Edge Cases

- 같은 항목에 동일 태그를 반복 입력하거나 공백·대소문자 차이만 있는 중복 태그를 입력한 경우, 중복으로 간주해 저장을 거부해야 한다.
- 공백만 있는 태그를 입력한 경우, 유효하지 않은 태그로 간주해 저장을 거부해야 한다.
- 태그가 5개 이하이더라도 일부 태그만 길이 제한을 초과하면 전체 저장을 거부해야 한다.
- 태그 비교 규칙은 앞뒤 공백을 제거한 뒤 대소문자를 구분하지 않는 방식으로 일관되게 적용되어야 한다.
- 허용되지 않은 문자(공백, 한글, 특수문자 등)가 포함된 태그는 저장을 거부해야 한다.
- 태그 필터 조회 시 저장된 항목은 많지만 조건 일치 항목이 없으면 빈 결과를 명확히 안내해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 새 ToDo 항목 생성 시 태그를 선택적으로 함께 입력할 수 있어야 하며, 입력 문법은 `--tag` 옵션 반복 방식으로 제공되어야 한다.
- **FR-002**: 사용자가 태그를 입력하지 않아도 ToDo 항목은 기존과 동일하게 생성될 수 있어야 한다.
- **FR-003**: 하나의 ToDo 항목에는 최대 5개의 태그만 연결할 수 있어야 한다.
- **FR-004**: 각 태그는 20자 이하의 텍스트여야 한다.
- **FR-005**: 공백만으로 이루어진 태그는 허용되지 않아야 한다.
- **FR-006**: 같은 ToDo 항목 안에서 동일한 태그를 중복 저장할 수 없어야 한다.
- **FR-007**: 태그 제한을 위반한 입력은 항목 저장 전체를 거부하고, 사용자가 수정할 수 있는 명확한 안내를 제공해야 한다.
- **FR-008**: 시스템은 ToDo 항목과 연결된 태그를 항목 데이터와 함께 유지해야 한다.
- **FR-009**: 사용자는 목록 조회 시 단일 태그를 기준으로 결과를 필터링할 수 있어야 한다.
- **FR-010**: 태그 필터 조회 결과에는 요청한 태그가 연결된 항목만 포함되어야 한다.
- **FR-011**: 요청한 태그와 일치하는 항목이 없으면 조건에 맞는 항목이 없다는 안내를 제공해야 한다.
- **FR-012**: 태그 기능이 추가된 이후에도 기존의 ToDo 추가, 전체 조회, 완료, 삭제 기능은 동일한 방식으로 동작해야 한다.
- **FR-013**: 기존에 저장된 태그 없는 ToDo 항목은 데이터 손실이나 동작 변경 없이 계속 조회, 완료, 삭제할 수 있어야 한다.
- **FR-014**: 태그의 유효성 검사와 필터 일치 판단은 앞뒤 공백을 제거한 뒤 대소문자를 구분하지 않는 동일 기준으로 처리되어야 한다.
- **FR-015**: 태그는 비교 시 정규화 규칙을 사용하더라도, 사용자에게 표시되는 값은 해당 태그의 최초 입력 원형을 유지해야 한다.
- **FR-016**: 태그는 영문자(a-z, A-Z), 숫자(0-9), 하이픈(-), 언더스코어(_)만 포함할 수 있어야 하며 공백은 허용되지 않아야 한다.

### Key Entities *(include if feature involves data)*

- **ToDo 항목**: 사용자가 관리하는 할 일 단위로, 제목, 상태, 선택적 마감 정보와 함께 0개 이상의 태그를 가진다.
- **태그**: ToDo 항목을 분류하기 위한 짧은 식별 텍스트로, 한 항목 안에서 중복 없이 최대 5개까지 연결된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 태그가 포함된 새 ToDo를 30초 이내에 생성할 수 있다.
- **SC-002**: 태그 입력 제한 위반 사례에서 100%의 경우 항목이 저장되지 않고, 사용자가 원인을 이해할 수 있는 안내가 표시된다.
- **SC-003**: 태그 기준 목록 조회는 검증용 데이터셋에서 100% 정확하게 일치 항목만 반환한다.
- **SC-004**: 태그를 사용하지 않는 기존 ToDo 사용 흐름의 핵심 시나리오(추가, 전체 조회, 완료, 삭제)는 변경 후에도 100% 성공한다.
- **SC-005**: 사용자는 조건에 맞는 태그 목록을 한 번의 조회 명령으로 확인할 수 있고, 조건 불일치 시 빈 결과를 명확히 구분할 수 있다.

## Assumptions

- 태그는 단일 항목에 부가되는 분류 정보이며, 프로젝트 전역 태그 사전이나 별도 관리 화면은 제공하지 않는다.
- 태그 필터링은 한 번에 하나의 태그 기준으로만 수행한다.
- 태그 텍스트는 입력 전후 불필요한 공백을 제거한 기준으로 검증된다.
- 대소문자 차이만 있는 태그는 같은 태그로 취급하며, 표시값은 최초 입력 원형을 유지한다.
- 태그 허용 문자는 영문/숫자/-/_로 제한하며, 공백과 기타 특수문자는 허용하지 않는다.
- 기존에 저장된 태그 없는 항목은 별도 변환 작업 없이 계속 사용할 수 있다.
